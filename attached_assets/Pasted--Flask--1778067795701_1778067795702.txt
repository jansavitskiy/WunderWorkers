Контекст
У меня есть веб‑приложение на Flask для учёта рабочего времени сотрудников. Уже работают: авторизация, роли (admin/employee), организации (с ИНН), типы работ, добавление отчётов сотрудниками, экспорт CSV, админ‑панель с просмотром отчётов. Сейчас при создании отчёта автоматически проставляется текущая дата и время (date_created). Нужно дать сотруднику возможность выбрать дату, за которую он отчитывается (например, отчёт создаётся 7 мая, а работа была выполнена 5 мая).

Цель
Сотрудник должен иметь возможность выбрать календарную дату (день, месяц, год) отдельно от даты фактического создания отчёта. Выбранная дата – это дата выполнения работы. Она должна сохраняться в базу и использоваться во всех отчётах и экспорте вместо даты создания отчёта (или наряду с ней).

Что нужно реализовать
1. Добавить поле в модель Report
Поле work_date (DateTime) – дата, за которую отчитывается сотрудник.

Значение по умолчанию – текущая дата (UTC) при создании, но пользователь может его изменить.

Поле date_created (DateTime) оставить как дату фактического заполнения отчёта (для аудита).

Почему два поля?

work_date – «за какой день работа» (важно для отчётов, зарплаты).

date_created – «когда отчёт написали» (например, запоздалое заполнение не меняет учётный период).

2. Обновить форму добавления отчёта для сотрудника
Добавить поле выбора даты (input type="date") с меткой "Дата выполнения работы".

По умолчанию подставлять сегодняшнюю дату.

Пользователь может выбрать любую дату в прошлом (максимум – сегодняшняя дата, будущие даты запретить).

Поле обязательно для заполнения.

3. Сохранение отчёта
При сохранении отчёта записывать выбранную work_date (дата начала дня, время можно установить 00:00:00).

date_created – автоматически текущее время (не редактируется).

4. Админ-панель: отображение и фильтрация
В разделе «Отчёт за день» добавить фильтр по диапазону дат выполнения работы (work_date).

По умолчанию фильтр показывает отчёты за сегодня (по work_date).

Добавить возможность фильтровать по work_date, а не по date_created.

В таблице отчётов показывать две даты: «Дата работы» (work_date) и «Дата создания отчёта» (date_created).

5. Экспорт в CSV
В CSV-файле должны быть две отдельные колонки:

Дата работы (формат ДД.ММ.ГГГГ)

Дата создания отчёта (формат ДД.ММ.ГГГГ ЧЧ:ММ)

Обе даты должны быть в московском времени (MSK, UTC+3).

Порядок колонок (пример):
Сотрудник;Организация;ИНН;Тип работы;Часов;Что выполнено;Дата работы;Дата создания

6. JS на стороне клиента
Запретить выбор даты в будущем (max – сегодня).

При выборе даты можно показывать предупреждение, если дата сильно в прошлом (опционально).

Обрабатывать локаль (формат даты – ГГГГ-ММ-ДД, стандарт input date).

Технические требования
Использовать UTC в базе данных, но для отображения и CSV – московское время (MSK).

Миграция БД: добавить столбец work_date в таблицу reports.

Все существующие отчёты при миграции должны получить work_date = date_created (дата создания) – чтобы не потерять старые записи.

В админ-панели фильтр по датам должен корректно работать с work_date (использовать report_date или work_date – но давайте назовём поле work_date).

Если поле в форме не заполнено – установить текущую дату.

Шаги для реализации (ожидаемый код)
Миграция – добавить поле work_date (DateTime, nullable=False, default=datetime.utcnow).

Обновить форму в time_report.html – добавить <input type="date" name="work_date" value="{{ today }}" max="{{ today }}">.

Обновить маршрут /time_report (POST) – извлечь work_date из формы, распарсить, сохранить в объект Report.

Обновить маршруты админ-панели:

/admin/daily_report – фильтрацию переделать с date_created на work_date (или добавить выбор).

Передавать в шаблон отчёты с двумя датами.

Обновить шаблон daily_report.html – добавить колонку «Дата работы» (и, возможно, «Дата создания»).

Обновить export_daily_report – добавить обе даты в CSV (сначала дату работы, потом дату создания).

Обновить функцию конвертации времени (to_msk, msk_strftime) – использовать для обеих дат.

Обновить модель Report – добавить поле work_date.

Что должно получиться в итоге
Сотрудник заходит в раздел «Заполнить отчёт», видит поле «Дата выполнения работы» с календарём.

Может выбрать любой день в прошлом (включая выходные и праздники).

Сохранённый отчёт в базе имеет корректную work_date (дату работы) и date_created (дату заполнения).

Админ в разделе «Отчёт за день» смотрит отчёты, сгруппированные по дате работы.

CSV экспорт показывает обе даты, что позволяет анализировать как период выполнения работ, так и оперативность заполнения отчётов.

Пожалуйста, напиши полный код для реализации этой функциональности, включая:

Модель Report с добавленным полем work_date и миграцию.

Обновлённую форму сотрудника (time_report.html) с выбором даты.

Обновлённый маршрут /time_report (сохранение work_date).

Обновлённый маршрут /admin/daily_report (фильтрация по work_date, отображение двух дат).

Обновлённый шаблон daily_report.html с новыми колонками.

Обновлённый экспорт CSV (export_daily_report) с двумя колонками дат.

JavaScript для ограничения выбора даты (не в будущее).

Конвертацию даты работы в московское время для отображения.

Весь код должен быть готов к использованию, с комментариями на русском языке.
